CPU vs GPU vs TPU

A systems-level lecture on processor architectures — from transistors to AI inference

Before we write a single line of code or run a single benchmark, we must understand what these chips are philosophically designed to do. Every architectural decision — pipeline depth, cache hierarchy, memory bandwidth — flows from this fundamental design intent.

CPU
Central Processing Unit

The generalist. Designed to execute any sequential task as fast as possible. Think of the CPU as a brilliant professor who can solve any problem — but only works on one (or a few) problems at a time. Its secret weapon: massive caches, branch prediction, and out-of-order execution.

GPU
Graphics Processing Unit

The parallel specialist. Originally for rendering pixels (the same shader operation across millions of pixels simultaneously), the GPU evolved into a massive SIMD machine. Think: thousands of simple workers executing the same instruction on different data at once.

TPU
Tensor Processing Unit

The domain-specific accelerator. Google engineered the TPU from the ground up for one thing: matrix multiply-accumulate operations — the core primitive of neural networks. It sacrifices generality entirely for peak throughput on this single operation class.

The Fundamental Trade-off (Hennessy & Patterson's insight):
Every processor design is a point on a graph between latency optimization (how fast can one task finish?) and throughput optimization (how many tasks can finish per second?). The CPU is at one extreme, the TPU at the other, and the GPU sits in between — but closer to the TPU end for ML workloads.

The architectural differences are staggering — and understanding them at a transistor/microarchitecture level is what separates a software engineer from a systems architect. Let's go deep.

The CPU: Out-of-Order Superscalar Execution

A modern CPU core (e.g., Intel Golden Cove, AMD Zen 4) contains: a 14–20 stage pipeline, a reorder buffer (ROB) holding 300–600 in-flight instructions, a branch predictor with ~99.9% accuracy, multiple execution ports (ALU, FPU, AGU, load/store), and up to 48 MB of L3 cache. The entire design philosophy is: minimize latency for a single thread of execution.

Key Insight: On a modern CPU, roughly 80–90% of the die area is devoted to control logic — caches, branch predictors, OoO machinery. Only 10–20% is actual arithmetic. The CPU is paying a massive silicon tax for flexibility.
// CPU sees this as ~6 dependent instructions // OoO engine finds hidden parallelism across iterations for (i = 0; i < N; i++) { c[i] = a[i] * b[i] + d[i]; // 1 FMA per cycle, superscalar } // CPU dispatches 2-4 such operations simultaneously via ILP

The GPU: SIMT — Single Instruction, Multiple Threads

NVIDIA's SIMT model organizes threads into warps of 32 threads. All 32 threads execute the same instruction simultaneously on different data. An H100 GPU has 132 Streaming Multiprocessors (SMs), each with 128 CUDA cores = 16,896 CUDA cores total, plus dedicated Tensor Cores for matrix ops.

The memory hierarchy is radically different: L1 cache is small (~256KB per SM), L2 is shared (~50MB), but HBM3 memory bandwidth hits 3.35 TB/s — enabling the GPU to feed thousands of cores without starving them.

Warp Divergence — The GPU's Achilles Heel: If threads within a warp take different branches (if-else), the GPU must execute both paths serially, masking threads that didn't take that path. This is why GPUs hate irregular control flow. Write predictable, branchless algorithms for maximum efficiency.

The TPU: Systolic Arrays — Pure Dataflow

This is where it gets elegant. Google's TPU uses a systolic array — a 2D grid of multiply-accumulate (MAC) units. For TPU v4, this is a 128×128 grid = 16,384 MACs, each performing a multiply and addition per cycle.

Here's the beauty: activations flow in from the left, weights flow in from the top, and partial sums flow downward. No cache misses, no branch prediction, no control overhead. Data literally pulses through the hardware like a heartbeat (hence "systolic" — from the Greek for heartbeat).

Why This Works for Neural Networks: A single transformer forward pass is dominated by matrix multiplications: Q·K^T, softmax(·)·V, FFN projections. A systolic array computes C = A·B with optimal arithmetic intensity: O(N³) compute for O(N²) memory — exactly the roofline sweet spot.

Now we reason like architects. We'll cover memory hierarchies, the Roofline Model, and why these differences fundamentally change how you write software.

The Memory Wall Problem

Since the early 1990s, CPU compute performance has grown at ~55%/year while DRAM bandwidth has grown at only ~23%/year. This gap is the Memory Wall. It's why modern CPUs have 32–48MB of L3 cache — they're desperately trying to hide memory latency.

GPUs attacked this differently: instead of caching, they use latency hiding through massive thread-level parallelism. While one warp waits for memory, 100 other warps are executing. TPUs attack it via the systolic array's data reuse: once a weight matrix is loaded, it's reused across thousands of input vectors before being evicted.

The Roofline Model

Every kernel (function) has an arithmetic intensity = FLOPs / bytes of memory traffic. The Roofline Model says: your performance is bounded by either peak compute OR peak bandwidth, whichever you hit first.

Arithmetic Intensity Examples:
• Vector addition (a[i] + b[i]): ~0.16 FLOP/byte → memory-bound on every chip
• Dense matrix multiply (N×N): ~N/2 FLOP/byte → compute-bound for large N
• Attention (transformer): ~O(seq_len) FLOP/byte → transitions between both regimes

TPUs are designed to keep matrix multiplies in the compute-bound regime. CPUs can't because their bandwidth is too high relative to their cache capacity.

Flynn's Taxonomy

Michael Flynn's 1966 taxonomy classifies processors by instruction and data streams:

SISD — Single Instruction, Single Data → Classical CPU (1 operation on 1 datum) SIMD — Single Instruction, Multiple Data → CPU vector units (AVX-512: 16 floats at once) MIMD — Multiple Instructions, Multiple Data → Multi-core CPU (each core independent) SIMT — Single Instruction, Multiple Threads → GPU (NVIDIA's extension of SIMD) DATAFLOW — No instruction stream at all → TPU systolic array (pure data movement)

Understanding this taxonomy tells you immediately why you can't just "put a neural net on a CPU and expect GPU speed." The computation models are fundamentally different at the hardware level.

Thermal & Power Efficiency (FLOPS/Watt)

The Data Center Equation: At scale, FLOPS/Watt is more important than raw FLOPS. Google's TPU pods deliver ~110 petaFLOPS/second per pod while consuming ~110kW — roughly 1 TFLOPS/W for BF16. This is why hyperscalers invest billions in custom silicon. Over 5 years, the TCO of compute is dominated by electricity, not hardware capex.

Interconnect: The Hidden Bottleneck

A single chip is never the full story. At scale, what limits you is how fast chips talk to each other. NVLink (NVIDIA) delivers 900 GB/s chip-to-chip bandwidth. Google's ICI (Inter-Chip Interconnect) for TPU pods delivers ~100 Gbps per link with a 3D torus topology. The CPU's PCIe 5.0 x16 delivers a paltry ~64 GB/s.

This is why, when training a 175B parameter GPT-3, the bottleneck isn't FLOPs — it's the all-reduce communication between 1,000+ GPUs/TPUs synchronizing gradients after every backward pass.

Architecture doesn't emerge in a vacuum. Each processor class was born from a specific engineering pressure. Understanding history is understanding inevitability.

1971
Intel 4004 — The First Microprocessor
2,300 transistors at 10μm. 4-bit operations. 108 kHz. The direct ancestor of every CPU you've ever used. Designed for Busicom calculators.
1993–1999
3D Graphics Accelerators Emerge
3dfx Voodoo, NVIDIA RIVA TNT — dedicated hardware for triangle rasterization. The GPU is born as a fixed-function pipeline, not programmable at all.
2001
NVIDIA GeForce 3 — Programmable Shaders
First GPU with programmable vertex and pixel shaders. Researchers immediately ask: "Can we run non-graphics computations on this?" The seed of GPGPU is planted.
2006
NVIDIA CUDA — The GPGPU Revolution
CUDA makes GPU programming accessible to non-graphics developers. Papers begin appearing: GPUs for protein folding, PDE solving, and signal processing. Throughput computing enters mainstream CS.
2012
AlexNet — Deep Learning Meets GPU
Krizhevsky, Sutskever, Hinton train AlexNet on 2× GTX 580 GPUs. Top-5 error drops 10 percentage points. Every AI lab buys GPUs. NVIDIA's stock begins a decade-long ascent.
2016
Google TPU v1 — The First Production AI ASIC
Deployed internally at Google, the TPU v1 handles inference for Google Search, Street View, and Translate. 92 TOPS (INT8) at 40W. Purpose-built systolic array. Published at ISCA 2017 — one of the most-cited architecture papers of the decade.
2017
NVIDIA Tensor Cores — GPUs Fight Back
The Volta V100 introduces Tensor Cores: dedicated 4×4 matrix multiply units. GPUs now have domain-specific hardware for ML, closing the gap with TPUs for training workloads.
2020–2024
The ASIC Arms Race
AWS Trainium/Inferentia, Meta MTIA, Microsoft Maia, Amazon Graviton, Apple Neural Engine, Cerebras WSE — every hyperscaler builds custom silicon. The GPU's monopoly on AI compute fractures. TPU v4 deployed in 4096-chip pods. H100 SXM5 at 989 TFLOPS BF16.
